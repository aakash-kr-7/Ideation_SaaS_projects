# Database Design

## Principles
- **Migration-first**: All schema changes are handled via `supabase/migrations/*.sql` and pushed using the Supabase CLI. Never make manual schema changes in the Dashboard.
- **Row Level Security (RLS)**: All 37 public tables have RLS enabled. Tenant-owned rows are isolated via `team_id`, `user_id`, or relations that resolve through a run/project/team. `scoring_weights` is deliberately global read-only reference data for authenticated users, so it is not a tenant table.
- **Foreign Keys & UUIDs**: All primary and foreign keys use UUIDs (`gen_random_uuid()`).

## Key Tables
- `users`: Extends `auth.users` with profile metadata. Created automatically via the `handle_new_user` Postgres trigger upon sign up.
- `teams` & `team_members`: Foundation for multi-tenant isolation. Every user gets a default team on creation.
- `projects`: High level folders containing multiple `research_runs`.
- `research_runs`: Represents a single validation run.
- `opportunities`, `evidence_items`, `competitors`, `risks`, etc.: Normalized outputs generated by the worker, strictly linked by foreign keys to the original `research_run`.
- `api_usage_logs`: Itemized provider attempts and estimated cost for a run.
- `reasoning_agent_outputs`, `report_versions`, and `report_exports`: Citation-validating specialist outputs, immutable report payloads, and stored export metadata.

## Storage
- `user-assets`: Public bucket for user avatars.
- `exports`: Private bucket for generated JSON, Markdown, CSV, and PDF reports.
- `cached-sources`: Private bucket for raw HTML scrapes to avoid re-fetching.
