# Database Design

## Principles
- **Migration-first**: All schema changes are handled via `supabase/migrations/*.sql` and pushed using the Supabase CLI. Never make manual schema changes in the Dashboard.
- **Row Level Security (RLS)**: Every table has RLS enabled. Tenant isolation is strictly enforced via the `team_id` or relations that resolve to `team_id`.
- **Foreign Keys & UUIDs**: All primary and foreign keys use UUIDs (`gen_random_uuid()`).

## Key Tables
- `users`: Extends `auth.users` with profile metadata. Created automatically via the `handle_new_user` Postgres trigger upon sign up.
- `teams` & `team_members`: Foundation for multi-tenant isolation. Every user gets a default team on creation.
- `projects`: High level folders containing multiple `research_runs`.
- `research_runs`: Represents a single validation run.
- `opportunities`, `evidence_items`, `competitors`, `risks`, etc.: Normalized outputs generated by the LLM workers, strictly linked by foreign keys to the original `research_run`.

## Storage
- `user-assets`: Public bucket for user avatars.
- `exports`: Private bucket for generated PDF/CSV reports.
- `cached-sources`: Private bucket for raw HTML scrapes to avoid re-fetching.
